Task 1: Step 2 Architecture Diagram

Layered Architecture

Controller Layer
  ├── UserController         ← register, changeEmail, changePassword, getUserById
  ├── AuthController         ← login, logout, refresh, verify
  ├── BillingController      ← subscribe, renew, cancel, getPlans, getPayments
  ├── ContentController      ← create, upload, update, delete, getContents
  └── WatchHistoryController ← recordWatch, getHistory

Service Layer
  ├── UserService           ← business logic ของ User, Profile, UserPreference
  ├── AuthService           ← business logic ของ AuthManager, UserSession
  ├── BillingService        ← business logic ของ Subscription, Plan, Payment
  ├── ContentService        ← business logic ของ Content, Video, Music
  └── WatchHistoryService   ← business logic ของ WatchHistory

Repository Layer
  ├── UserRepository        ← query users table
  ├── ProfileRepository     ← query profiles table
  ├── UserPreferenceRepository ← query user_preferences table
  ├── SessionRepository     ← query sessions table
  ├── PlanRepository        ← query plans table
  ├── SubscriptionRepository← query subscriptions table
  ├── PaymentRepository     ← query payments table
  ├── VideoRepository       ← query contents table WHERE type = Video
  ├── MusicRepository       ← query contents table WHERE type = Music
  └── WatchHistoryRepository← query watch_history table

Domain Layer
  ├── User, Profile, UserPreference
  ├── UserSession, AccessPrincipal
  ├── Plan, Subscription, Payment
  ├── SubscriptionStatus, PaymentStatus
  ├── Content, Video, Music
  └── WatchHistory

Task 2: Step 3 Contract

ก่อนจะอ่านหลังจากนี้ขอบอกไว้ก่อนว่า ทุก endpoint ต้องมี Authorization header ยกเว้นแค่ 3 เส้นนี้เท่านั้น
1. POST /api/identity/users     ← สมัครสมาชิก ยังไม่มี token
2. POST /api/auth/login         ← login ยังไม่มี token
3. POST /api/billing/payments/webhook ← Stripe/Omise เรียกมา ใช้ Signature แทน

Client ส่ง request มา
Header: Authorization: Bearer "eyJ..."
      │เมื่อย
      ▼
Middleware ดึง token ออกมาจาก header
      │
      ▼
AuthService.verify("eyJ...")
      │
      ├── token หมดอายุ    → ปฏิเสธ 401 Unauthorized
      ├── token ถูก revoke → ปฏิเสธ 401 Unauthorized
      │
      └── token ปกติ → สร้าง AccessPrincipal
                        { userId: "u-123", ... }
                              │
                              ▼
                        ส่งต่อให้ Controller
                        และ Service ใช้งานต่อ

Table Design

กลุ่ม IDENTITY
1) users
- userId       STRING  PK
- email        STRING  UNIQUE
- passwordHash STRING
- createdAt    TIMESTAMP

2) profiles
- profileId   STRING  PK
- userId      STRING  FK → users
- displayName STRING
- avatarUrl   STRING

3) user_preferences
- prefId    STRING  PK
- userId    STRING  FK → users
- prefKey   STRING
- prefValue STRING


กลุ่ม AUTH
4) user_sessions                      ← มาจาก UserSession
- sessionId    STRING    PK
- userId       STRING    FK → users
- accessToken  TEXT      UNIQUE
- refreshToken TEXT      UNIQUE
- expiresAt    TIMESTAMP

INDEX: accessToken  ← ใช้ทุก request
INDEX: userId       ← ค้นหา session ของ user

[AuthManager]      → ไม่มี table เป็น Service stateless
[SessionRepository]→ ไม่มี table เป็น Repository คุยกับ user_sessions
[AccessPrincipal]  → ไม่มี table สร้างใน RAM ชั่วคราวตอน verify()
                     ดึงข้อมูลจาก user_sessions + subscriptions


กลุ่ม BILLING
5) plans                              ← มาจาก Plan
- planId       STRING    PK
- name         STRING
- price        FLOAT
- billingCycle STRING

6) subscriptions                      ← มาจาก Subscription
- subscriptionId STRING    PK
- userId         STRING    FK → users
- planId         STRING    FK → plans
- status         STRING    ← "PENDING" / "ACTIVE" / "EXPIRED" / "CANCELLED"
- startDate      TIMESTAMP
- endDate        TIMESTAMP

[SubscriptionStatus] → ไม่มี table เก็บเป็น status column ใน subscriptions
                       PENDING / ACTIVE / EXPIRED / CANCELLED

7) payments                           ← มาจาก Payment
- paymentId             STRING    PK
- subscriptionId        STRING    FK → subscriptions
- externalTransactionId STRING    UNIQUE
- status                STRING    ← "PENDING" / "COMPLETED" / "FAILED" / "REFUNDED"
- amount                FLOAT
- currency              STRING
- paidAt                TIMESTAMP
- deletedAt             TIMESTAMP

[PaymentStatus] → ไม่มี table เก็บเป็น status column ใน payments
                  PENDING / COMPLETED / FAILED / REFUNDED


กลุ่ม CONTENT
8) contents                           ← มาจาก Content (Abstract)
- contentId STRING    PK
- creatorId STRING    FK → users
- type      STRING    ← "Video" หรือ "Music"
- title     STRING
- cdnUrl    TEXT

9) videos                             ← มาจาก Video
- contentId  STRING    PK, FK → contents
- resolution STRING
- duration   INT
- codec      STRING

10) music                             ← มาจาก Music
- contentId STRING    PK, FK → contents
- artist    STRING
- bitrate   INT
- genre     STRING

11) watch_history                     ← มาจาก WatchHistory
- historyId       STRING    PK
- userId          STRING    FK → users
- contentId       STRING    FK → contents
- watchedAt       TIMESTAMP
- watchedDuration INT
